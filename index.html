The Problem

Why can't we just bazel build?


Dependency sprawl
S-CORE pulls from Bazel module registry, PyPI, crates.io, GitHub release URLs — dozens of external hosts during a single bazel fetch //….
Network restrictions
Corporate networks, air-gapped CI, and developer machines behind proxies all struggle with unconstrained egress.
Non-reproducibility
External hosts can change, go offline, or serve different content. A build that passes today may fail tomorrow.
Slow cold starts
Without a shared action cache, every CI runner re-compiles the same artifacts from scratch.

The Solution: score-cache

One container. Four servers. Zero external traffic.


score-cache is a self-hosted, hermetic dependency mirror and build cache assembled entirely from tools already native to the S-CORE toolchain — no commercial artifact repository required.

  • Single OCI image, tagged score-cache:YYYYMMDD-<graph-hash>
  • All four servers behind one hostname (score-cache)
  • Each server speaks the exact protocol the build tool expects — no client shims
  • Every dependency downloaded once, sha256-verified, stored immutably
  • Published to GHCR; old tags never deleted → historical builds are reproducible forever

What's Inside the Container

All four managed by supervisord inside the container.

Why a Public Cache Container?


Open-source by default
S-CORE is developed in the open. Contributors need to get a working build without any internal infrastructure.
Zero-configuration onboarding
Pull the image, start it, add three lines to .bazelrc. No accounts, no licenses, no VPN required.
Reproducibility guarantee
A public immutable tag means any contributor can rebuild any historical version of the codebase.
Ecosystem composability
The same image works as a Dagger service, a Docker Compose dependency, or a Kubernetes sidecar — because it is just an OCI image.

Configuring Your Build Tools

Three snippets — one per ecosystem

.bazelrc

build --remote_cache=http://score-cache:9092
build --remote_timeout=60
build --experimental_remote_cache_async

.cargo/config.toml

[source.crates-io]
replace-with = "score-cache"

[source.score-cache]
registry = "sparse+http://score-cache:8081/cargo/index/"

pip.conf / pyproject.toml

[[tool.uv.index]]
name    = "score-cache"
url     = "http://score-cache:3141/root/pypi/+simple/"
default = true

In Docker Compose or a Dagger service network, the hostname score-cache resolves automatically. For local use set SCORE_CACHE_HOST=localhost.

In a Corporate Environment

score-cache and Artifactory are interchangeable backends


score-cache (open-source)
  • Self-hosted OCI image
  • bazel-remote / devpi / nginx
  • Versioned with your lock files
  • Zero cost, open license
Artifactory / Nexus (centralized)
  • Managed service or on-prem
  • Bazel Remote Cache repo
  • PyPI Proxy repo
  • Cargo / Generic repo

The build tools are unchanged. Only the URL in the config switches. A corporate team can replace score-cache with Artifactory by editing three config snippets — no Bazel rules, no Python code, no Cargo source changes.

Built with Dagger

The entire cache lifecycle is a Dagger pipeline


1
resolve-graph — runs bazel fetch //… inside a proxied Dagger container; extracts every URL + sha256 from lock files into url-inventory.json
2
seed-cache — downloads & verifies every dependency; populates bazel-remote CAS, devpi, cargo-mirror, and nginx artifact store into image layers
3
verify-hermetic — runs full build & test in a network-isolated Dagger container; fails if any external request escapes to the internet
4
publish-cache — pushes tagged image to GHCR; mirrors to long-retention archive registry
5
record-manifest — appends immutable digest + lock fingerprints to versions.json in source control

Dagger Pipeline Code

Pipelines are typed Python functions — not shell scripts

@function
async def score_cache(self) -> dagger.Container:
    """Build, verify, and publish the score-cache image."""
    inventory  = await self.resolve_graph()        # 1
    seeded     = await self.seed_cache(inventory)  # 2
    _verified  = await self.verify_hermetic(seeded)# 3
    digest     = await self.publish_cache(seeded)  # 4
    await self.record_manifest(digest)             # 5
    return seeded

  • Each stage is a Dagger Container — fully reproducible, cache-aware
  • The verify-hermetic stage uses Dagger's service-binding API to allow only score-cache egress
  • dagger call score-cache runs the full pipeline; dagger call score-cache-plan dry-runs it

Why Dagger?


Portable pipelines
The same Dagger function runs identically on a developer laptop, in GitHub Actions, and in any other CI provider — no YAML rewrites.
Native service binding
Dagger can start score-cache as a service and bind the build container to it with one call, with all other egress blocked — no iptables, no network namespaces to manage.
Typed, testable DAGs
Pipelines are Python @function methods. Pure logic is unit-tested with pytest. Dagger stages are integration-tested with dagger call.
Container-native caching
Dagger caches every layer automatically. Adding a new Python package re-runs only the devpi seed stage, not the entire pipeline.

Dagger in score/reference-integration

How Dagger helps across the S-CORE ecosystem


Polyrepo orchestration — config/repos.yaml is a catalog of all S-CORE repos. A single dagger call integration runs build + test across all of them, selecting repos by tag, profile, or name.

score-cache as a Dagger service — Any pipeline stage that needs hermetic resolution can bind score-cache as a Dagger service. The stage gets a guaranteed-available cache with zero external traffic — no Docker Compose, no Kubernetes, no side-channel setup.

reference-integration gate — The verify-hermetic stage proves that the full S-CORE build tree (fetch + build + test) passes with only score-cache reachable. This becomes the reference-integration acceptance criterion.

Dagger Modules

Reusable pipeline building blocks


  • score-cache as a shared Dagger module — any repo in the S-CORE ecosystem can consume score-cache as a Dagger module dependency and get a ready-to-use cache service with one import.
  • Composable stages — resolve-graph, seed-cache, and verify-hermetic are individually callable. A downstream module can call only the stages it needs.
  • Version-locked services — modules pin a specific score-cache image tag. A Dagger module update is the only way to change which cache image a pipeline uses — full auditability.
  • Local and CI parity — developers run dagger call score-cache-plan locally and get the exact same execution graph CI will run — including the same cache image tag.

Decisions Taken

Choices that are settled and in effect


done Mirror, not vendor
Rust crates are served from an internal sparse registry mirror, not checked in as source. Mirrors are immutable per image tag.
done Date + graph-hash tag format
YYYYMMDD-<graph-hash> makes image provenance and freshness immediately legible. Hash changes only when lock files change.
done versions.json is append-only
Once an entry is written it is never modified. Digest + fingerprint records are immutable facts in source control.
done supervisord manages all four servers
One container, one process supervisor. No sidecar containers, no Kubernetes pods required for local use.
done Dagger Python SDK
Pipeline logic is typed Python @function methods, not YAML or shell. Pure helpers are unit-tested with pytest via make test.
done sha256 required for every entry
Entries without a public sha256 are logged as warnings and skipped by the seeders. No unverified content enters the cache.

Open Decisions

Questions still being worked through


open OCI registry sidecar
Should score-cache include a local OCI distribution registry on port 5000 for base-image hermeticity? Adds complexity; needed only if container base images must be fetched offline.
open Read-only vs. read-write cache in CI
bazel-remote defaults to read-write. Should CI push action results, or should the seeded image be the only write path? Trade-off between cache warmth and reproducibility.
open Long-retention archive registry
GHCR with no-delete policy, or a separate org-controlled registry? Policy and cost trade-offs under discussion.
open Baselibs integration timing
When to land the .bazelrc lines in eclipse-score/baselibs (Phase 5). Blocked on hermetic verification passing for the full baselibs build graph.
open Vendoring as hardening path
Checking crate sources into the repo is an option for maximum air-gap hardening. Currently out of scope but not ruled out.
open Dagger module publication
Publishing score-cache as a reusable Dagger module to the Dagger module registry — enables any S-CORE downstream to consume it as a typed dependency.

Summary


  • score-cache solves S-CORE's dependency availability, reproducibility, and build speed problems with one OCI image.
  • It is a drop-in replacement for Artifactory in open-source environments — same protocols, same client config, different URL.
  • Dagger makes the pipeline portable, testable, and service-network-aware — hermetic verification is a first-class feature, not an afterthought.
  • Every image is immutable, versioned, and archived — historical builds are reproducible indefinitely.
  • Open decisions are tracked and bounded; the architecture is ready to grow into a shared Dagger module for the wider S-CORE ecosystem.
